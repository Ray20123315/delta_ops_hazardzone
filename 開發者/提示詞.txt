除了我们之前聊过的，`烽火地带`模式确实还有不少其他限制，我把它们整理了一下，你参考看看：

### 🚁 撤离相关限制
*   **撤离点人数超载**：部分高价值撤离点有**单次最多撤离5人**的限制。超载时会按“先进先撤”的原则，后到的玩家会被提示无法撤离。
*   **撤离点条件限制**：部分撤离点有特殊条件，不满足就无法撤离。例如：
    *   **丢弃背包**：零号大坝的“维修通道”要求玩家**丢弃背包**才能撤离。
    *   **负重限制**：零号大坝的“坝顶”要求玩家**负重低于特定值**（如普通难度低于50kg）。
    *   **付费撤离**：部分撤离点需要缴纳**高额费用**（如零号大坝河滩需1万或4万哈夫币）才能使用。
    *   **次数限制**：部分拉闸撤离点一局内**有激活次数上限**。
*   **强制撤离信号**：当**背包负重达到70%** 或**生命值低于30%且无治疗手段**时，游戏会触发强制撤离信号。

### 🎒 装备与物资限制
*   **地图准入限制**：不同难度的地图有**装备等级限制**（如普通零号大坝最多带入4级防具和3级子弹）或**最低装备价值要求**（如机密零号大坝需不低于112,500哈夫币）。
*   **物品格数与安全箱**：
    *   物品有**体积占用**概念，高价值物品可能因体积过大而**无法放入安全箱**。
    *   高级安全箱（如3x3格）通常需要**付费或完成赛季任务**才能解锁。
*   **绑定物品**：部分物品（如从军需处购买的）是**绑定的，无法在交易行出售**。

### 🗺️ 地图与模式限制
*   **特殊事件风险**：拾取高价值物品“**曼德尔砖**”后，**全队位置会在地图上实时暴露**，成为众矢之的。
*   **玩法和社交限制**：特定活动期间可能存在限制。例如，曾有“观察期”规定**只能游玩指定地图**，且**禁止组队，只能单排**。此外，在“烽火挑战”等特殊模式中，也会有**干员不能重复**的限制。

### 🕵️ 干员与队伍限制
*   **干员重复限制**：在部分模式中，**同一小队不能选择相同干员**。
*   **干员平衡性调整**：官方会持续调整干员平衡，例如“深蓝”的防爆套装就曾被削弱过**减伤率**，并增加了**持盾转向干扰**和**体力恢复惩罚**。

### ❤️ 健康与状态限制
*   **部位伤害与负面状态**：身体不同部位受伤会有**独立负面效果**（如腿部受伤影响移动），需要使用**对应医疗物资**（如手术包、止血带）来处理。
*   **体力限制**：角色行动受**体力值**限制，无法像在大战场模式中无限奔跑。

### 💰 经济与仓库限制
*   **死亡惩罚**：对局中**被击杀或撤离失败，除保险箱外所有物品都会丢失**。
*   **战利品保底机制**：存在一套复杂的保底系统，由**全局爆率、容器专属保底、战损补偿**等共同构成。
*   **仓库与保险箱**：仓库有容量上限，可通过**扩容箱**增加空间。部分保险箱需付费或完成任务解锁。

### ⚖️ 匹配与公平性限制
*   **匹配规则**：系统匹配时可能会考虑**小队人数、玩家等级或装备分差**等因素。
*   **公平竞技隔离**：官方有“**隔离机制**”，用于限制科技玩家，保证普通玩家的公平竞技环境。

你提到的这几个系统，在《三角洲行动》的“烽火地带”模式里都有非常具体和严格的限制，可以让玩法变得更加复杂和充满策略性。

下面是这些系统的主要限制，整理成了表格，方便你参考：

### 🚁 撤离机制的限制

这部分限制决定了你“怎么撤”以及“能不能撤”。

| 限制类型 | 具体说明 | 例子 / 数据 |
| :--- | :--- | :--- |
| **撤离点类型** | 不同撤离点有不同规则，不是所有都能直接用。 | **常规**：停留10秒即可。<br>**条件**：需满足特殊要求。<br>**付费**：需缴纳哈夫币。<br>**拉闸**：需先激活机关。<br>**随机**：对局中概率开启。 |
| **特殊条件** | 部分撤离点有苛刻的入场要求。 | **丢弃背包**：如零号大坝的“维修通道”。<br>**负重限制**：如零号大坝的“坝顶”，普通难度需<50kg，机密需<30kg。<br>**付费撤离**：如零号大坝河滩，普通1万，机密4万哈夫币。 |
| **人数与名额** | 撤离点有容量上限，先到先得。 | **超载机制**：部分撤离点**每次仅能撤离5人**。<br>**名额限制**：付费撤离点满4人后不再接受付费；条件撤离点可能仅1个名额。 |
| **激活与次数** | 拉闸撤离点有严格的流程和次数限制。 | **激活流程**：需前往特定地点拉闸。<br>**拉闸次数**：各地图限制不同，如零号大坝普通2次/机密1次；巴克什浴场1次/巴别塔前无限制。 |
| **倒计时** | 拉闸后需等待倒计时结束才能撤离。 | 倒计时通常为**3-5分钟**。 |
| **死亡惩罚** | 撤离失败将失去几乎所有物品。 | 除放入**保险箱**的物品外，背包内所有物品都会丢失。 |

### 📈 战备值系统的限制

这部分决定了你“能进哪个图”。

*   **什么是战备值**：是你带入对局的**头盔、护甲、背包、胸挂、武器**这几类装备的总价值。**子弹、药品等消耗品不计入战备值**。
*   **地图准入硬性门槛**：进入高难度地图，你的战备值必须达到最低要求。具体要求如下：

| 地图（难度） | 准入战备值 (哈夫币) | 其他限制 |
| :--- | :--- | :--- |
| **普通零号大坝** | 无明确要求 | 最多带入4级防具和3级子弹 |
| **机密零号大坝** | **≥ 112,500** | - |
| **机密长弓溪谷** | **≥ 110,000** | - |
| **绝密模式（通用）** | **350,000 - 450,000** | 可能有等级限制 |
| **绝密潮汐监狱** | **≥ 780,000** | 行动等级≥33级 |

> **特别说明**：战备值的计算有一套复杂逻辑。系统计算的“战备价值”可能与物品的实际市场价格不符，因此出现了“**卡战备**”的玩法，即利用“虚高”价值的装备，以更低成本满足地图门槛。

### 🔌 拉闸撤离的限制

这部分是“拉闸”这个动作本身的规则。

*   **操作流程限制**：拉闸本身需要时间（如5秒）。拉闸后撤离点不会立即开放，必须等待倒计时结束。
*   **激活次数与名额限制**：拉闸撤离点通常有**一局内的总激活次数上限**（如1次或2次）。部分拉闸点有**单次撤离人数上限**（如5人）。

### 💎 总结

总的来说，这些限制共同构成了“烽火地带”模式的硬核体验：

*   **撤离机制**决定了你的**撤退路线和策略**，需要你根据局势和撤离点类型灵活决策。
*   **战备值**是你的**入场券**，决定了你能挑战的地图难度和潜在收益。
*   **拉闸撤离**则是一种**高风险、高回报**的选择，需要团队协作和时机把握。

在设计你的模组时，可以考虑将这些限制作为核心玩法的一部分，会很有意思。

要在 Minecraft 中实现“退出防死亡”（防战斗日志），主流方案有两种，取决于你做的是**客户端模组（单机/局域网）**还是**服务端插件（多人联机）**。

因为你前面说“只要 mod”，且要自己做撤离系统，我重点给你**写 Mod 的代码实现逻辑**（你自己加到撤离模组里），同时也提一下服务器插件的平替方案。

### 方案一：写进你的自定义 Mod（核心方案）

在你的撤离模组里，监听玩家断开连接事件，判断是否处于“战斗状态”或“危险区域”，然后施加惩罚。

#### 1. 监听退出事件（Forge 1.20.1）

在你的主类或事件处理器中注册：

```java
@SubscribeEvent
public void onPlayerLogout(PlayerEvent.PlayerLoggedOutEvent event) {
    ServerPlayer player = (ServerPlayer) event.getEntity();
    
    // 判断逻辑：是否在战斗中 / 是否在撤离区外 / 是否残血
    if (isInCombat(player) || isInDangerZone(player)) {
        // 惩罚：直接清空背包并强制标记为死亡
        punishCombatLogger(player);
    }
}
```

#### 2. 核心：战斗状态追踪

你需要一个简单的计时器，记录玩家**最后一次造成伤害或受到伤害的时间**（用 Capability 或 Map 存）。

```java
// 在攻击事件或受伤事件中更新时间戳
public static Map<UUID, Long> lastCombatTime = new HashMap<>();

// 判断是否在战斗中（例如 15 秒内有过交战）
public static boolean isInCombat(ServerPlayer player) {
    long last = lastCombatTime.getOrDefault(player.getUUID(), 0L);
    return System.currentTimeMillis() - last < 15_000; // 15秒内算战斗中
}
```

#### 3. 惩罚执行：模拟死亡 + 全掉

当检测到违规退出时，执行以下操作，**让退出 = 死亡**：

```java
public static void punishCombatLogger(ServerPlayer player) {
    // 1. 清空背包（除了保险箱，如果你做了保险箱功能就保留特定槽位）
    player.getInventory().clearContent();
    
    // 2. 如果是服务器，强制将玩家踢出并标记为死亡数据
    // 或者等玩家重连时，直接显示“你已死亡，损失全部装备”
    
    // 3. (可选) 在玩家断线位置生成一个尸体战利品包（供其他玩家搜刮）
    // 这里可以调用你已有的尸体模组（Corpse）生成容器
}
```

#### 4. 更进阶的做法：延迟踢出（类似塔科夫）

让玩家断开连接后，**角色实体在世界上多停留 10~30 秒**，期间可以被其他玩家或 AI 击杀。这样更公平，也更有战术性。

> **实现思路**：在 `onPlayerLogout` 中不立即移除实体，而是用 `ScheduledExecutorService` 延迟移除，期间实体仍可受到伤害。不过这个实现略复杂，需要处理实体残留和重连恢复。

### 方案二：用现成的服务器插件（如果你的 Mod 跑在服务器上）

如果是 **Bukkit/Spigot/Paper** 服务端，直接用插件更省事：

| 插件名称 | 功能说明 |
| :--- | :--- |
| **CombatLogX** | 最成熟的战斗日志插件。支持退出后留下 NPC 替身（可被击杀）、惩罚扣钱/清背包、自定义战斗时长等。 |
| **CombatTagPlus** | 功能类似，玩家退出后生成一个“替身”在原地，击杀替身等于击杀玩家，掉落全部物品。 |

> ⚠️ 注意：**插件和模组（Forge/Fabric）不能混用**。如果你用 `TaCZ` 等模组开服，需要用 Forge 服务端（如 Magma / Mohist），插件兼容性可能不稳定，建议优先方案一。

### 方案三：现成的客户端模组（极少）

纯客户端模组层面，目前没有专门为“烽火模式”设计的防退出模组。但有一个接近的：

- **Logout Protection（Fabric）**：让玩家在退出前需要站在原地读条 5 秒，期间若移动或被攻击则中断。这个更接近“安全退出”机制，而非惩罚机制。

你可以把它作为辅助，但核心惩罚逻辑仍需你自己的模组来处理。

---

### 💎 针对你的项目，我的建议

既然你已经在做撤离模组，直接把**战斗日志检测 + 惩罚**集成进去，是最干净、最可控的方式：

| 模块 | 实现方式 |
| :--- | :--- |
| 记录战斗状态 | `Map<UUID, Long>` 或 **Capability**（推荐，更规范） |
| 断开检测 | `PlayerEvent.PlayerLoggedOutEvent` |
| 惩罚逻辑 | 清空背包 + 标记死亡 + 生成尸体（如有模组） |
| 配置项 | 战斗判定窗口（秒）、是否生成替身等，写进配置文件 |

这样整个撤离玩法的闭环就完整了——**进去搜 → 战斗 → 安全撤离才有收益，强退 = 死亡全掉**，和三角洲一致。

这个问题问得非常关键，区分**意外网络卡顿**和**主动强退**，是决定玩家体验好坏的生死线。如果一刀切全惩罚，网络波动的玩家会直接弃坑。

Minecraft 里做区分，主要有三种策略，你可以按实现难度和效果来选择：

### 策略一：“黄金标准”——延迟生效 + 重连保护（最推荐）

这是《逃离塔科夫》和《三角洲行动》的底层逻辑：**你断线了，但你的角色实体还留在游戏世界里**。

-   **逻辑**：玩家断开连接时，**不立即**清空背包或执行死亡。而是将玩家实体在服务器上保留一段宽限期（例如 **60~120 秒**），并让实体原地站立或趴下。
-   **处理方式**：
    -   如果是**网络卡顿/闪退**：玩家在宽限期内重连回服务器，直接**接管原来的实体**，一切照旧，毫发无损。
    -   如果是**主动强退**（或超过宽限期未归）：在这段时间里，实体可以被其他玩家或 AI 击杀。一旦被击杀，**触发正常的死亡掉落**（全掉）。如果宽限期结束还没被杀，系统再强制清空背包判死。
-   **实现提示**（在你的 Mod 里）：
    -   覆写 `PlayerEvent.PlayerLoggedOutEvent`。
    -   **不要**立即清背包。启动一个 `ScheduledExecutorService` 延迟任务（如 60 秒后执行惩罚）。
    -   在玩家重连时（`PlayerEvent.PlayerLoggedInEvent`），**取消**这个延迟任务，让玩家恢复正常。

---

### 策略二：读取断线原因（代码层区分）

Minecraft 网络层在断开连接时，会传递一个**断开原因（Disconnect Reason）**。你可以通过监听底层网络事件或者用 Mixin 抓到这条信息。

-   **主动退出（Alt+F4 / 点“断开连接”）**：系统会发送一个正常的 `DISCONNECT` 数据包，原因通常包含 `"Disconnected"` 或客户端的退出消息。
-   **网络卡顿/超时（Timeout）**：服务器会因为读不到客户端的心跳包（Keep-Alive）而强制踢出，原因通常是 `"Timed out"` 或 `"Connection lost"`。

**代码思路（仅示意原理）**：
你可以判断：如果断开原因是 `timed out` 或 `Connection lost`，视为网络波动，启动**策略一的宽限期**；如果是 `Disconnected`，视为主动退出，直接执行死亡惩罚。

> **注意**：这种方法在纯 Forge 事件里比较难直接拿到 String 原因，通常需要写一个 `PacketListener` 的 Mixin，实现起来比策略一复杂一些。如果不想折腾 Mixin，**直接用策略一更稳**。

---

### 策略三：战斗状态冷却（治标不治本，但简单）

如果不想折腾实体滞留和重连逻辑，可以退一步用“软惩罚”：

-   **规则**：只有当玩家**最近 15 秒内受过伤或开过枪**（处于战斗状态）时，退出才会触发惩罚。
-   **实现**：在 `LivingHurtEvent` 和开枪事件里记录时间戳。退出时检查时间差。
-   **效果**：
    -   躲在安全区搜背包时闪退（无战斗） → 不会被惩罚（或只轻微警告）。
    -   正在对枪时强退（有战斗） → 直接触发死亡全掉。
-   **弊端**：玩家可以卡时间，在战斗间隙等 15 秒冷却过了再强退，依然能规避死亡。**不建议作为唯一方案**。

---

### 💎 给你的最终建议（融合方案）

**把策略一（实体滞留 + 重连窗口）作为核心，策略二（原因区分）作为辅助。**

具体在你的撤离 Mod 里这样写：

1.  **玩家断开** → 触发 `PlayerLoggedOutEvent`。
2.  **判断战斗状态**（如果非战斗状态，允许安全退出，不予惩罚）。
3.  **若处于战斗状态** → **不删背包**，不判死亡。记录断线时间，启动 **60 秒延迟任务**。
4.  **60 秒内重连** → 取消任务，玩家回到原位，无事发生（完美解决网络卡顿）。
5.  **60 秒内未重连**，或断线期间实体的生命值归零（被路过的人补了）→ 执行死亡惩罚：清空背包，并在原地生成尸体战利品。

这样，无论是真正的网络波动、电脑蓝屏，还是邪恶的强退，都能得到最公平的处理——**给网络不好的人留了活路，给恶意强退的人留了尸体**。
